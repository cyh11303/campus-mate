# Campus Mate DB Persistent Version

Campus Mate는 캠퍼스 내 **밥약 신청**, **음식 갤러리 공유**, **룸메이트 매칭** 기능을 제공하는 웹 서비스입니다.

기존 메모리 기반 저장 방식을 SQLite 데이터베이스 기반 영속 저장 방식으로 확장했으며, Railway 배포 환경에서는 SQLite DB 파일과 업로드 파일을 Volume에 저장하여 재배포 후에도 데이터가 유지되도록 구성했습니다.

---

## 실행 방법

### 로컬 실행

```bash
pip install -r requirements.txt
uvicorn server:app --reload
```

### 로컬 접속 주소

```text
http://127.0.0.1:8000
```

### 로컬 DB 생성

* 실행하면 프로젝트 폴더에 `campus_mate.db`가 자동 생성됩니다.
* 업로드된 이미지는 로컬의 `uploads/` 폴더에 저장됩니다.

---

## Railway 배포 환경

Railway에서는 SQLite DB와 업로드 파일을 영구 저장하기 위해 Volume을 사용합니다.

### Railway 환경 변수

```text
DATABASE_PATH=/data/campus_mate.db
UPLOADS_DIR=/data/uploads
```

### Railway 저장 구조

* SQLite DB 파일: `/data/campus_mate.db`
* 업로드 이미지 폴더: `/data/uploads`
* Railway Volume Mount Path: `/data`

즉, Railway 자체 DB를 사용하는 것이 아니라, **SQLite 파일을 Railway Volume에 저장하는 방식**입니다.

## 주요 페이지 구성

### `index.html`

* 기본 메인 홈페이지입니다.
* 사용자가 `/` 주소로 접속하면 표시되는 진입 페이지입니다.
* `https://web-production-3a6e6.up.railway.app/`로 접속했을 때 동작하는 기본 페이지입니다.

### `main.html`

* `index.html`과 동일한 메인 화면 복사본입니다.
* 개발 과정에서 `/main.html`로도 메인 화면에 직접 접근할 수 있도록 유지한 파일입니다.
* 현재 기능상 `index.html`과 큰 차이는 없습니다.

### `meal.html`

* 밥약 신청 시스템 페이지입니다.
* 밥약 신청, 신청 현황 조회, 밥약 요청/수락/거절, 요청함 확인 기능을 포함합니다.
* 현재 실제 갤러리 기능도 `meal.html` 내부의 갤러리 탭에서 동작합니다.

### `roommate.html`

* 룸메이트 매칭 시스템 페이지입니다.
* 학번 기반 해시 저장, 조건 필터링, 가중 유사도 계산, 캐시 확인, 룸메 요청/수락/취소 기능을 포함합니다.

### `gallery.html`

* 독립 갤러리 페이지 파일입니다.
* 다만 현재 서비스의 주요 사용 흐름에서는 `meal.html` 내부의 갤러리 탭을 통해 갤러리 기능을 사용합니다.
* 즉, 현재 실제 갤러리 기능은 `meal.html` 안에서 작동합니다.

## DB 구성

### `users`

* 룸메이트 등록 정보 저장

### `trait_weights`

* 룸메이트 생활 성향별 가중치 저장

### `room_requests`

* 룸메이트 신청 상태 저장
* 신청, 수락, 거절, 취소 요청, 취소 완료 상태를 관리합니다.

### `meals`

* 밥약 신청 정보 저장
* 학번, 이름, 메뉴, 연락수단, 연락수단 세부정보 등을 저장합니다.

### `meal_requests`

* 밥약 요청 정보 저장
* 요청자, 대상자, 메시지, 요청 상태 등을 저장합니다.

### `gallery_posts`

* 음식 갤러리 게시글 저장
* 이미지 경로, 작성자, 메뉴, 태그, 리액션, 삭제 비밀번호 해시값 등을 저장합니다.

## 밥약 시스템

### 1. 밥약 신청

사용자는 다음 정보를 입력하여 밥약 신청자로 등록할 수 있습니다.

* 학번
* 이름
* 먹고 싶은 메뉴
* 연락수단
* 연락수단 세부정보

등록된 신청자는 신청 현황 목록에 표시됩니다.

### 2. 개인정보 보호

학번은 개인정보 보호를 위해 전체를 그대로 표시하지 않습니다.

예시:

```text
2024****01
```

### 3. 신청 현황 조회

밥약 신청 현황에서는 등록된 신청자 목록을 확인할 수 있습니다.

각 신청자 카드에는 다음 정보가 표시됩니다.

* 사용자 프로필 아이콘
* 이름
* 마스킹된 학번
* 먹고 싶은 메뉴
* 밥약 예정 인원
* 같이 먹기 요청 버튼

기존 음식 아이콘은 실제 메뉴와 맞지 않을 수 있어 제거했고, 공통 사용자 프로필 아이콘으로 변경했습니다.

### 4. 같이 먹기 요청

신청 현황에서 다른 사용자에게 `같이 먹기 요청`을 보낼 수 있습니다.

요청은 바로 확정되지 않고, 먼저 `pending` 상태로 저장됩니다.

### 5. 요청함 기능

사용자는 본인의 학번을 입력하여 밥약 요청함을 확인할 수 있습니다.

요청함은 다음 두 영역으로 나뉩니다.

* 받은 요청
* 보낸 요청

받은 요청은 수락하거나 거절할 수 있습니다.
보낸 요청에서는 내가 보낸 요청의 상태를 확인할 수 있습니다.

### 6. 수락/거절 기능

받은 요청을 수락하면 요청 상태가 `accepted`로 변경됩니다.
거절하면 `rejected`로 변경됩니다.

요청이 수락되면 신청 현황의 `밥약 예정 인원`이 증가합니다.

예시:

```text
밥약 예정 1명
밥약 예정 2명
```

### 7. 연락수단 공개

밥약 신청자의 연락수단은 신청자 목록에 바로 공개하지 않습니다.

* 요청이 수락되기 전: 연락수단 비공개
* 요청이 수락된 후: 요청을 보낸 사람이 상대방 연락수단 확인 가능

### 8. 밥약 새로고침

밥약 화면의 새로고침 버튼을 누르면 다음 동작이 수행됩니다.

* 신청 현황 다시 조회
* 요청함 학번 입력 초기화
* 받은 요청/보낸 요청 영역 초기화

## 갤러리 시스템

갤러리는 현재 `meal.html` 내부의 갤러리 탭에서 동작합니다.

### 1. 음식 사진 업로드

사용자는 다음 정보를 입력하여 음식 사진을 업로드할 수 있습니다.

* 이름
* 메뉴
* 해시태그
* 삭제 비밀번호 4자리
* 이미지 파일

### 2. 삭제 비밀번호

무분별한 삭제를 방지하기 위해 사진 업로드 시 삭제 비밀번호 4자리를 입력하도록 했습니다.

게시글 삭제 시에는 업로드할 때 입력한 비밀번호와 동일한 4자리를 입력해야 삭제할 수 있습니다.

### 3. 비밀번호 저장 방식

삭제 비밀번호는 원문으로 저장하지 않습니다.

* 입력값: 숫자 4자리
* 저장 방식: SHA-256 해시값 저장

### 4. 기존 DB 마이그레이션

기존 SQLite DB에 `delete_password_hash` 컬럼이 없는 경우를 대비하여, 서버 실행 시 자동으로 컬럼을 추가하도록 마이그레이션 코드를 넣었습니다.

```sql
ALTER TABLE gallery_posts ADD COLUMN delete_password_hash TEXT DEFAULT ''
```

기존 비밀번호가 없는 게시글은 삭제가 제한될 수 있습니다.

## 룸메이트 매칭 시스템

### 1. 룸메이트 정보 등록

사용자는 다음 정보를 입력해 룸메이트 매칭 시스템에 등록할 수 있습니다.

* 학번
* 이름
* 학과
* 성별
* 본인 흡연 여부
* 평균 기상 시간
* 평균 취침 시간
* 연락수단
* 연락수단 세부정보
* 생활 성향 체크리스트
* 생활 성향별 가중치

### 2. 해시 테이블 저장

룸메이트 등록 정보는 학번 기반 해시 함수로 계산된 슬롯에 저장됩니다.

해시 테이블 크기는 소수인 31을 사용합니다.

```text
hash[31]
슬롯 0 ~ 30
```

해시 함수는 문자열 누적 해시 방식을 사용합니다.

```text
h = (h × 37 + charCode[i]) % 31
```

해시 테이블 크기 31과 multiplier 37을 다르게 두어, 특정 슬롯으로 값이 몰리는 문제를 줄였습니다.

### 3. 1차 필터링

매칭 검색 시 모든 사용자를 바로 비교하지 않고, 먼저 절대 조건을 기준으로 후보를 줄입니다.

절대 조건은 다음과 같습니다.

* 성별 동일
* 흡연 여부 동일

즉, 남자는 남자끼리, 여자는 여자끼리만 매칭됩니다.
흡연자는 흡연자끼리, 비흡연자는 비흡연자끼리만 매칭됩니다.

### 4. 2차 유사도 계산

1차 필터링을 통과한 후보를 대상으로 생활 성향과 생활 시간을 비교합니다.

유사도 계산에는 다음 요소가 반영됩니다.

* 생활 성향 체크리스트
* 성향별 가중치
* 평균 기상 시간
* 평균 취침 시간

### 5. 가중치 범위

성향 가중치는 기본 1.0에서 최대 1.5까지 설정할 수 있습니다.

```text
가중치 범위: 1.0 ~ 1.5
```

가중치가 유사도를 과도하게 흔들지 않도록 최대값을 1.5로 제한했습니다.
성향 선택 여부가 매칭의 핵심 기준이 되고, 가중치는 중요도를 보정하는 역할을 합니다.

### 6. 제거된 성향 항목

평균 기상 시간과 평균 취침 시간 입력값과 의미가 겹치는 항목은 제거했습니다.

* 아침형 인간
* 야행성

시연 및 설명의 명확성을 위해 다음 항목도 제거했습니다.

* 손님 초대 OK

### 7. 캐시 기능

같은 학번으로 반복 검색할 경우, 이전 결과를 캐시에 저장해 빠르게 반환합니다.

동작 흐름은 다음과 같습니다.

```text
Cache MISS → 후보 계산 → 결과 저장
Cache HIT → 저장된 결과 즉시 반환
```

## 룸메이트 신청 규칙

룸메이트 신청은 다음 상태로 관리됩니다.

* 대기 상태
* 룸메 예정 상태
* 취소 요청 상태
* 취소 완료 상태

### 1. 대기 상태

대기 상태는 룸메 신청을 보냈지만 상대가 아직 수락하거나 거절하지 않은 상태입니다.

이 상태에서는 아직 룸메가 확정된 것이 아니므로 다음이 가능합니다.

* 여러 사람에게 신청 가능
* 다른 사람도 나에게 신청 가능

### 2. 룸메 예정 상태

상대가 신청을 수락하면 두 사람은 룸메 예정 상태가 됩니다.

룸메 예정 상태가 되면 다음이 제한됩니다.

* 내가 다른 사람에게 새 신청 불가
* 다른 사람이 나에게 새 신청 불가
* 이미 룸메 예정인 상대와 중복 신청 불가

즉, 한 명과 룸메 예정이 되면 그 사용자는 잠긴 상태가 됩니다.

### 3. 취소 요청 및 취소 완료

룸메 예정 상태는 한쪽이 일방적으로 취소할 수 없습니다.

취소 흐름은 다음과 같습니다.

```text
한쪽이 취소 요청
→ 상대가 취소 수락
→ 취소 완료
```

취소가 완료되면 다시 다른 사람에게 신청할 수 있습니다.

## 배포 주의사항

SQLite 파일과 업로드 파일은 로컬에서는 프로젝트 폴더에 저장되지만, Railway 같은 배포 환경에서는 재배포 시 사라질 수 있습니다.

이를 방지하기 위해 Railway에서는 Volume을 사용합니다.

```text
SQLite DB: /data/campus_mate.db
업로드 파일: /data/uploads
```

Railway에서 다음 환경 변수를 사용합니다.

```text
DATABASE_PATH=/data/campus_mate.db
UPLOADS_DIR=/data/uploads
```

Railway Volume을 삭제하면 DB와 업로드 이미지도 함께 삭제될 수 있으므로 주의해야 합니다.

## 배포 후 확인 방법

Railway에 배포한 뒤에는 다음 주소에서 기능을 확인할 수 있습니다.

```text
https://web-production-3a6e6.up.railway.app/
```

주요 확인 주소:

```text
/               → 메인 페이지
/meal.html      → 밥약 신청 및 갤러리 탭
/roommate.html  → 룸메이트 매칭
/api/list       → 밥약 신청자 목록 JSON
/api/gallery    → 갤러리 게시글 목록 JSON
/table          → 룸메이트 해시 테이블 상태
```

배포 후 화면이 바로 바뀌지 않으면 브라우저 캐시 문제일 수 있으므로 `Ctrl + F5`로 강력 새로고침합니다.

또는 주소 뒤에 버전 쿼리를 붙여 확인할 수 있습니다.

```text
/meal.html?v=latest
/roommate.html?v=latest
```
